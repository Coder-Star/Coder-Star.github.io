## 跨域原理及解决

**相关原理**

跨域问题是在浏览器中出现的情况，主要原因是因为浏览器本身的同源策略机制的限制，同源策略（Same Origin Policy，简称SOP问题）是一种安全策略；

URL= 协议 + 域名 + 端口 + 协议；如果两个 URL 的协议、域名和端口相同，则它们同源，如果不相同，则不同源，则就会引起跨域；

浏览器会限制脚本中发起的跨域请求。比如，使用 XMLHttpRequest 对象和 Fetch 发起 HTTP 请求就必须遵守同源策略。不允许跨域访问并非是限制浏览器发起请求，而是返回的结果被浏览器给拦截了，接收不到；最好的例子是 CSRF 跨站攻击原理，无论是否跨域，服务器都接收到了请求。

有些浏览器不允许从 HTTPS 跨域访问到 HTTP，比如 Chrome 和 Firefox，这些浏览器在请求还未发出的时候就会拦截请求,这是特例。

**解决方案**

### CROS

这是方式需要浏览器与服务器的支持，目前所有浏览器都支持该功能，并且对我们是透明的，不用关注，只要我们的服务端实现了 CROS 通信，就可以进行跨源通信。简单来说，就是服务端在Response中添加了一些响应头，前端校验通过后就允许非同源可以正常通信了。

CORS 将请求分为了两类，分别为简单请求和非简单请求。
简单请求会直接发出 CROS 请求，会在 头信息 中增加一个 Orignin 字段。
非简单请求会在实际请求之前，发送一个 OPTIONS 方法的预检请求向服务器确认，通过之后，才会发送真正的请求，避免跨域请求对服务器的数据的用户数据造成影响。同样的 url 调用多次只会在第一次调用的时候发送预检请求，后面无需发送。

其中简单请求要求包括

1. 请求方法是以下三种方法之一：

- HEAD
- GET
- POST

2. HTTP 的头信息不超出以下几种字段：

- Accept
- Accept-Language
- Content-Language
- Last-Event-ID
- Content-Type：只限于三个值 application/x-www-form-urlencoded、multipart/form-data、text/plain

不是简单请求的都是非简单请求。

Request 里关于跨域的头信息包括

- Access-Control-Request-Headers
- Access-Control-Request-Method

Response 里关于跨域的头信息包括

- Access-Control-Allow-Origin  该字段必填。它的值要么是请求时Origin字段的具体值，要么是一个*，表示接受任意域名的请求
- Access-Control-Allow-Methods 该字段必填。它的值是逗号分隔的一个具体的字符串或者*，表明服务器支持的所有跨域请求的方法。注意，返回的是所有支持的方法，而不单是浏览器请求的那个方法。这是为了避免多次"预检"请求
- Access-Control-Allow-Headers 该字段可选。CORS请求时，XMLHttpRequest对象的getResponseHeader()方法只能拿到6个基本字段：Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragma。如果想拿到其他字段，就必须在Access-Control-Expose-Headers里面指定
- Access-Control-Allow-Credentials 该字段可选。它的值是一个布尔值，表示是否允许发送Cookie.默认情况下，不发生Cookie，即：false。对服务器有特殊要求的请求，比如请求方法是PUT或DELETE，或者Content-Type字段的类型是application/json，这个值只能设为true。如果服务器不要浏览器发送Cookie，删除该字段即可
- Access-Control-Max-Age 该字段可选，用来指定本次预检请求的有效期，单位为秒。在有效期间，不用发出另一条预检请求。如果发现每次请求都伴随OPTIONS请求，则需要配置该字段，避免每次都发送预检请求。

简单请求中，当 Origin 指定的源不在服务器许可范围内，则不会返回`Access-Control-Allow-Origin`这个字段，浏览器就会抛出相应的错误，提示无法跨域请求。如果在服务端许可范围内，则服务器会返回`Access-Control-Allow-Origin`、`Access-Control-Allow-Headers`以及
`Access-Control-Allow-Credentials`字段。

非简单请求中，先发送的预检请求会附带`Access-Control-Request-Headers`以及`Access-Control-Request-Method`头信息用于表示即将的正式请求中会携带的头信息，服务器收到请求后，会校验`Origin`、`Access-Control-Request-Headers`以及`Access-Control-Request-Method`三个字段后，确认是否允许跨域请求。等通过了预检请求后，后面的正式请求与简单请求类似。

`chrome://flags/#out-of-blink-cors`

```java
// spring mvc中设置跨域的关键代码

//全局配置方式1，不过WebMvcConfigurer已经被标记为Deprecated
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOrigins("*")
                .allowedMethods("GET", "HEAD", "POST", "PUT", "DELETE", "OPTIONS")
                .allowCredentials(true)
                .maxAge(3600)
                .allowedHeaders("*");
    }
}

//全局配置方式2，基于过滤器Filter
@WebFilter(filterName = "CorsFilter ")
@Configuration
public class CorsFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) throws IOException, ServletException {
        HttpServletResponse response = (HttpServletResponse) res;
        response.setHeader("Access-Control-Allow-Origin","*");
        response.setHeader("Access-Control-Allow-Credentials", "true");
        response.setHeader("Access-Control-Allow-Methods", "POST, GET, PATCH, DELETE, PUT");
        response.setHeader("Access-Control-Max-Age", "3600");
        response.setHeader("Access-Control-Allow-Headers", "Origin, X-Requested-With, Content-Type, Accept");
        chain.doFilter(req, res);
    }
}

// 局部方式
public class GoodsController {
  // 可以放在method以及class上面

  @GetMapping("goods-url")
  public Response queryGoodsWithGoodsUrl(@RequestParam String goodsUrl) throws Exception {

  }
}  

// 这上面三种配置方式如果都用，会遵循就近原则。
```
