推送相关

发表于 2021-01-28 更新于 2021-08-21 分类于 iOS ， 相关 阅读次数：
本文字数： 3.3k 阅读时长 ≈ 3 分钟

iOS 推送分为本地推送、在线推送、离线推送。

  • 本地推送就是本地可以通过固定时间以及地点等条件触发推送操作。
  • 在线推送就是 APP 在前台利用长链接等形式实现推送，如 QQ、微信在前台时收到的消息；
  • 离线推送是指通过 APNs 服务器发送通知到 APP。

离线推送

推送流程基本如下：

  1. APP 申请推送权限
  2. APP 申请获得 deviceToken
  3. APNs 服务器验证成功后，将 deviceToken 返回至 APP 端
  4. APP 端将 deviceToken 存储至服务器，服务器带着证书以及 deviceToken 发送到 APNs 服务器
  5. APNs 服务器下发推送消息到 APP 端

APP 申请推送权限的时候会将设备的 device id 以及 APP 的 bundle id 以及证书等上报至 APNs 服务器，APNs 服务器会根据这两个信息生成 deviceToken。但是 device id 和 bundle id 不会确定唯一的 device token，对于同一部手机，卸载后重装 APP 的 device token 是会发生变化的，而且老的 token 不会无效，还可以正常推送。这应该是苹果的一个 Bug。

相关回调方法

一般需要静默推送，需要开启 Background Modes 里面的 Remote notifications 选项。静默推送满足条件如下：

  • 不能加 alert
  • 加”content-available” : 1
  • 不加 sound

iOS 新增了NotificationServiceExtension以及NotificationServiceExtension。

NotificationServiceExtension：当系统收到通知后会唤醒它，并允许它修改通知内容，之后再展示通知。如果想唤起 NotificationServiceExtension，payload 必须符合这样几个条件：

  • 必须增加 mutable-content 字段并为 1，这表示允许客户端修改这个通知；
  • 这个通知必须展示一个 alert，如果只是一个修改 badge 的通知的话，是不会唤起 service extension 的；
  • 静默推送是不能唤起 service extension 的，所以 payload 中不能有”content-available” : 1 字段；

使用场景：

  • 可以实现 push 到达率统计功能
  • 可以实现不启动 APP，下载小文件功能
  • 可以进行 push 信息脱敏处理功能
  • 可以更改提示音功能
